# Readdy Demo Gap Analysis v1（母本附录 I / 验证项 V-001）

- **审计对象：** Readdy 平台生成前端 `project-11645859.zip`（Vite 8 + React 19 + Tailwind 3 SPA，14 页面，约 14,200 行 TS/TSX，`npm run build` 一次通过）
- **审计基准：** 母本 v2.1 —— 6.2 信息架构、6.9 页面规格、6.10 八态、6.12 组件矩阵、6.13 一致性组件、第 7 部分需求 ID、packages/contracts 契约与 fixtures、PDR-001
- **审计方式：** 8 组页面并行逐文件审计（2026-07-03）
- **V-001 状态：** 逐页映射完成，本文即映射产物。母本附录 I 的「待评审」占位由本文替代；Readdy 动态预览的像素级核对可选、不阻塞。

## 一、总体判定

**可作 M0 原型底座（全模拟、一次性），不可作需求来源，更不可直接演进为生产前端。**

| 判定 | 数量 | 含义 |
| --- | --- | --- |
| 修改后保留 | 46 | 布局/交互壳/视觉可留，数据模型与治理层按契约重灌 |
| 重做 | 19 | 信息架构或语义方向性错误，仅视觉素材可参考 |
| 缺失需新建 | 20 | 母本规定的页面/区域在 Demo 中不存在 |
| 删除 | 1 | 功能对比 tab（混淆平台自身竞品与客户的市场竞争对手） |

**四层系统性结论（全部 8 组一致）：**

1. **视觉与骨架层（可复用 30-40%）**：深色 glass-card 设计系统 token 化且全局统一；次级导航五项与母本 6.2.2 逐字一致；Today 双列、战役五区、互动三栏、洞察全屏栅格等布局骨架与母本页面规格方向吻合；八处「AI 建议 + 置信度 + 分类徽章」卡片是证据可见原则的良好起点。
2. **治理层（为零，全部需建）**：ActionProposal→Policy→Approval 链路、ExecutionAuthorization/Dry Run、Evidence Drawer / Cost Badge / Policy Badge、数据权利（allowed_actions）、审计语义、八态页面状态——在全部 14 个页面中均不存在。Demo 的「一键发布」「直接回复」「导出到剪贴板」均与硬边界 1 冲突。
3. **数据模型层（与契约全面错位）**：三级结果链（Qualified Lead → SAO → Verified Outcome）在全部页面零出现；状态枚举与 11 个状态机不一致（如 Lead 5 态 vs 契约 8 态、Campaign 4 态 vs 11 态）；单一黑盒分数替代六维评分（违反 LED-006）；「机会」卡实为 Signal；Campaign 被做成项目管理看板（违反 D-009）。**修复路径明确：mock 全部换成 packages/contracts/fixtures（术语、枚举、行业已对齐）。**
4. **需求级错误（不得沿用）**：mock 文案含「LinkedIn 私信自动发送」「从 LinkedIn 抓取潜客」（违反 D-019/硬边界 7）；平台清单预设了未关闭的 OPEN DECISION #3；归因默认「多触点」超出 ANA-003 的 MVP 模型集合；转化直推外部 CRM 为主库（违反 ENG-011）；广告投放口径（ROAS/CPM）主导洞察页（违反 1.6 非目标）。

## 三、术语与品牌统一修正清单（M0 改造时一次性处理）

1. 产品名 `GrowthOS`（含 index.html、canonical growthos.ai、localStorage key）→ Global Growth Workspace 待定品牌位。
2. 全站引入三级结果链术语与北极星 SAO 指标；删除「MQL」「转化线索」等旧漏斗词。
3. 状态/枚举/评分/意图分类全部对齐 `packages/contracts`（11 个状态机 + Schema 枚举）。
4. mock 数据全部替换为 contracts/fixtures（光伏/建材 × 东南亚/非洲，与 PDR-001 一致；Demo 现为北美 SaaS 叙事）。
5. 删除 D-019 违规文案（LinkedIn 自动私信/抓取）；渠道清单标注 OPEN DECISION #3 待定。
6. 一级导航 7 项 → 母本 6.2.1 上限 6 项（「AI 发布」并回「内容」）。
7. 角色词表对齐 membership 契约 8 角色枚举。
8. 模型厂商名（GPT-4o 徽章）→ 网关模型别名 + 成本（ADR-007）。

## 四、技术底座结论与 M0 复用策略

- **技术栈**：Vite 8 + React 19 SPA 与母本 11.2「Next.js」基线不符。**裁决：接受用于 M0**——M0 是全模拟一次性原型（母本 13.2），验证的是交互与业务闭环，不是生产架构；**M1 生产 App Shell 按 ADR 在 Next.js 重建**，本原型只输出设计令牌、组件形态、mock 数据形状，不迁移路由与构建体系。
- **工程清理（小改）**：删除未使用的 firebase / supabase / stripe 依赖；清理 vite.config 中 Readdy 平台残留变量；index.html 四个字体/图标 CDN 收敛；图标库收敛为一个；i18n 死脚手架处理。
- **代码位置**：导入 monorepo 为 `apps/prototype`（`@ggw/prototype`），README 标注「M0 一次性原型」定位与本文引用。

## 五、与母本/契约的双向修正

审计同时发现两处需要回写的非前端问题：

1. **contracts 缺 content/publish/engage 域 Schema**（现有 workspace/knowledge/lead/campaign/opportunity 五域）。M0 的内容/发布/互动页面可先用 mock 形状，M1 前须补 CRT/PUB/ENG 契约（受保护路径，走 plan mode）。
2. 母本 6.5 引导流程与 Demo 的 `/goal` 一次性向导定位冲突已按母本裁决：目标启动器常驻 PG-001，引导是价值路径（官网→提取→确认→首批产物）而非配置表单。
